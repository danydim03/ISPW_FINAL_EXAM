Exception Chaining: Quando chiami super(message, cause) o new DAOException(msg, e), l'eccezione originale viene preservata nella catena.

Stack Trace completo: e.printStackTrace() stampa l'intera storia: CreaOrdineException ‚Üí DAOException ‚Üí SQLException

Debugging: Puoi risalire alla root cause con e.getCause().getCause() fino a trovare null

Regola fondamentale: Le eccezioni tecniche (SQLException, IOException) NON devono mai raggiungere il layer Boundary.

Ogni layer traduce l'eccezione nel proprio vocabolario semantico, nascondendo i dettagli implementativi ai layer superiori.

üìã Note di Implementazione - Habibi Shawarma

  • üîó
    Exception Chaining Corretto
    Tutti i costruttori in HabibiException chiamano super(message, cause). Esempio: throw new DAOException("Errore accesso dati", e)
  • üìù
    Logging Centralizzato nella Boundary
    Il logging avviene solo in CreaOrdineGUIController con logger.severe(). I DAO non loggano, solo rilanciano.
  • üé≠
    Facade Pattern
    CreaOrdineFacade semplifica l'interfaccia per la GUI ma non modifica le eccezioni - le rilancia direttamente.
  • üë§
    Messaggi User-Friendly
    La GUI usa e.getMessage() per mostrare messaggi comprensibili. Mai esporre dettagli tecnici come SQLException all'utente.
  • ‚úÖ
    Checked vs Unchecked
    HabibiException estende Exception (checked) per forzare la gestione esplicita. Tutte le eccezioni dell'app sono checked.
  • üóÇÔ∏è
    Catch Multipli nella GUI
    La GUI usa multi-catch: catch (DAOException | MissingAuthorizationException e) per gestire pi√π tipi con lo stesso handler.
